## Project Testing Overview

The project uses **two separate testing stacks** — one for the backend and one for the frontend. No Vite was used for testing; all tests use **Jest** as the test runner.

```
Tests at a glance
══════════════════════════════════════════════════════
 Backend  │ Vitest (unit) + Jest+Supertest (integration)
 Frontend │ Jest + React Testing Library + MSW (mock API)
══════════════════════════════════════════════════════
```

---

## 1. Backend Tests

### How to Run
```bash
cd backend

# Run unit tests only
npm run test:unit

# Run integration tests only (requires PostgreSQL on port 5433)
npm run test:integration

# Run both
npm test
```

### Unit Tests — **Vitest**
**Location:** `backend/tests/unit/`
**Framework:** Vitest (not Jest — intentional choice for speed with ESM)
**Who wrote them:** Manually written in a prior session

These test individual controller and service functions in complete isolation using `vi.mock()` to replace the database. No real DB connection is made.

| Test File | What It Tests | # Tests |
|---|---|---|
| `epic4/workflow.controller.test.js` | Visit state machine: open/close visits, assign care team, get visit | ~15 |
| *(other unit test files)* | Auth, EMR, consent controllers | varies |

**Example — what a unit test checks:**
```js
// Ensures that closing an already-closed visit returns 400
it('should return 400 for already closed visit', async () => {
  mockDb.query.mockResolvedValueOnce({ rows: [{ state: 'closed' }] });
  await closeVisit(req, res);
  expect(res.status).toHaveBeenCalledWith(400);
});
```

---

### Integration Tests — **Jest + Supertest**
**Location:** `backend/tests/integration/`
**Framework:** Jest + Supertest (fires real HTTP requests against the Express app)
**Database:** Real PostgreSQL test DB (`healthcare_test_db` on port 5433)
**Who wrote them:** Manually written in a prior session

These tests spin up the actual Express server and hit real API routes. They verify end-to-end request → controller → database → response flows.

**Key difference from unit tests:** No mocks. Hits a real DB. Validates actual SQL, auth middleware, and RBAC enforcement.

| Area Tested | Route Example | What It Checks |
|---|---|---|
| Auth | `POST /api/auth/login` | Valid credentials return JWT |
| EMR records | `GET /api/emr/patients/:id/medical-records` | RBAC enforcement, data shape |
| Consent | `POST /api/consent/grant` | DB write, response format |
| Workflow | `PATCH /api/visits/:id/state` | State transition rules enforced |

## 2. Frontend Tests

### How to Run
```bash
cd frontend

# Run all tests (unit + integration via MSW) — one command
npm test -- --watchAll=false

# Run with coverage report
npm test -- --watchAll=false --coverage

# Run a specific test file
npm test -- --watchAll=false --testPathPattern="Login"

# Run in watch mode (re-runs on file save — great for development)
npm test
```

### Framework & Setup
| Tool | Role |
|---|---|
| **Jest** (via CRA) | Test runner, assertions |
| **React Testing Library** | Render components, query DOM |
| **MSW v1 (Mock Service Worker)** | Intercepts HTTP calls inside Jest — no real backend needed |
| **jest.polyfills.js** | Provides `TextEncoder`/`TextDecoder` for Node 18 jsdom compatibility |

---

### Test Files — All Written Manually

**Total: 7 test suites, 27 tests** — *all written by AI assistant, none auto-generated by any framework.*

> CRA generates one empty `App.test.js` by default. That original file was replaced with a proper component import test. Everything else was created from scratch.

---

#### `src/App.test.js` — Smoke Test
**Type:** Unit  
**What it tests:** The root `App` component can be imported without crashing (catches module-level errors like the `import.meta.env` bug we fixed in `researchApi.js`)

```
✓ App component imports correctly
```

---

#### `src/api/__tests__/authApi.test.js` — Auth API Service
**Type:** Unit  
**Mocking:** `apiClient` is fully mocked with `jest.mock`  
**What it tests:** That `authApi` functions call the right endpoints with the right payloads

```
✓ login() — calls POST /auth/login with emailOrPhone + password
✓ login() — returns server response data
✓ login() — propagates error on failure
✓ registerPatient() — calls POST /auth/register/patient with correct data
✓ verifyOTP() — sends email payload when input looks like an email
✓ verifyOTP() — sends phone payload when input looks like a phone number
```

---

#### `src/api/__tests__/emrApi.test.js` — EMR API Service
**Type:** Unit  
**Mocking:** `apiClient` is fully mocked  
**What it tests:** EMR API functions hit correct endpoints

```
✓ createMedicalRecord() — calls POST /emr/medical-records with record data
✓ getPatientMedicalRecords() — calls GET /emr/patients/:id/medical-records with params
✓ uploadLabResult() — calls POST /emr/lab-results with multipart/form-data header
```

---

#### `src/context/__tests__/AuthContext.test.js` — Auth Context Provider
**Type:** Unit  
**Mocking:** `authApi` and `tokenManager` utilities mocked  
**What it tests:** The React context that manages auth state throughout the app

```
✓ No user when no token exists on mount
✓ Fetches and sets user when a valid token exists
✓ Clears user and removes token when token verification fails
```

> These tests cover the **session persistence logic** — e.g. if you refresh the page, does your login survive? Does a tampered/expired token get cleared?

---

#### `src/routes/__tests__/ProtectedRoute.test.js` — Route Guard
**Type:** Integration  
**Mocking:** `AuthContext.Provider` used directly with controlled values  
**What it tests:** The `ProtectedRoute` component that gates pages behind auth

```
✓ Shows "Loading..." while auth state is initializing
✓ Redirects to /login when not authenticated
✓ Renders children when authenticated (no role restriction)
✓ Renders children when authenticated AND role is in allowedRoles
✓ Redirects to /unauthorized when role doesn't match allowedRoles
✓ Handles roleName property (case-insensitive normalization)
```

> This is critical for security — it proves that a patient can't access `/doctor/dashboard` etc.

---

#### `src/pages/auth/__tests__/Login.test.js` — Login Page (Integration)
**Type:** Integration  
**Mocking:** MSW intercepts real `fetch`/`axios` calls at network level  
**What it tests:** Full login user flow from form input → API call → context update

```
✓ Renders email, password fields and Sign In button
✓ Shows success message from router navigation state (e.g. after registration)
✓ Successful login: calls AuthContext.login() with user + token from server
✓ Failed login (401): shows error message, does NOT call AuthContext.login()
```

> The MSW handler for the success case is defined in `src/mocks/handlers.js` and returns `{ accessToken: 'fake-jwt-token', user: { email: 'doctor@test.com', role: 'doctor' } }`.

---

#### `src/pages/patient/__tests__/PatientDashboard.test.js` — Patient Dashboard (Integration)
**Type:** Integration  
**Mocking:** MSW provides all 5 API endpoints the dashboard fetches on mount  
**What it tests:** Data fetching, rendering, and reactive display of dashboard data

```
✓ Renders "Patient Hub" header on mount
✓ Displays active consent count (e.g. "2 GRANTED") from API
✓ Displays health facts (title + description) from API
✓ Displays activity history entries from API
✓ Shows "No active visits in queue" message when visits API returns []
```

> These are the most realistic tests — they verify that the whole pipeline from `useEffect` fetch → state update → DOM render works end-to-end without a real backend.

## 3. CI/CD Integration

### GitHub Actions Pipelines

| Pipeline | File | Triggers |
|---|---|---|
| Backend CI | `.github/workflows/backend-ci.yml` | Push/PR to `main`, `develop` |
| Frontend CI | `.github/workflows/frontend-ci.yml` | Push/PR to `main`, `develop` |

### What Each Pipeline Does

**Backend CI jobs:**
1. Spins up a PostgreSQL 15 service container
2. Applies all schema SQL files
3. `npm run test:unit` (Vitest, no DB)
4. `npm run test:integration` (Jest+Supertest, real DB)
5. Uploads coverage to Codecov
6. Docker build verification

**Frontend CI jobs:**
1. `npm ci` (clean install)
2. Lint check
3. **`npm test -- --watchAll=false --coverage`** ← runs all 27 tests
4. Uploads coverage to Codecov
5. `npm run build` (production bundle)
6. Build size analysis
7. Docker build verification

> The frontend pipeline requires **no database** because MSW mocks all API calls at the network layer inside Jest itself.
